Section 01

System Architecture Overview

N.E.X.A Loop is a pnpm monorepo with three Docker Compose services: postgres, api (NestJS on :3001), and web (Next.js on :3000). Server-side Next.js fetches use API_URL=http://api:3001 for container-to-container routing; browser fetches use NEXT_PUBLIC_API_URL.

Section 02

Authentication & Session Flow

JWT stored as an httpOnly cookie. Payload: { sub, email, orgId }. The token is stateless — no Redis session store required. Expiry: 7 days. All protected endpoints require the cookie; the API guard verifies the signature and attaches req.user.

Fig 2.1 — Swimlane: Registration & protected request auth flow across Browser → Middleware → API → DB
Security invariant: orgId is extracted from the verified JWT payload via @CurrentOrg(). It is never accepted from the request body on any protected endpoint. Any attempt to send orgId in the body is silently stripped by whitelist: true on the global ValidationPipe.

Registration — step-by-step

1
User submits registration form
React Hook Form validates client-side. Submits POST /auth/register with JSON body.
{ email, password, orgName, name? }
2
Global ValidationPipe checks RegisterDto
IsEmail(), MinLength(8) on password, MinLength(2) on orgName. Unknown fields stripped. 400 thrown on failure before the handler runs.
3
Prisma $transaction: Org → User → UserOrg
All three inserts are atomic. If any fails the entire transaction rolls back. Password hashed with bcrypt (10 rounds). Role set to OWNER.
tx.organization.create() → tx.user.create() → tx.userOrganization.create()
4
JWT signed → Set-Cookie httpOnly
Payload: { sub, email, orgId }. 7-day expiry. Cookie: httpOnly, sameSite: lax, secure in production. passwordHash is never included in the response (stripped via sanitizeUser()).
5
Frontend redirects to /dashboard
All subsequent requests include the cookie automatically. Middleware verifies cookie presence on every /dashboard/* route via the matcher pattern.
Section 03

Multi-Tenancy Data Model

Every entity table carries an orgId FK. All service methods scope every query to orgId extracted from the JWT. The database enforces foreign key constraints; the service layer enforces scope. No query can leak cross-tenant data.

Fig 3.1 — Organisation-scoped entity hierarchy. Every entity carries orgId as a non-nullable FK.
Service-layer pattern: Always findFirst({ where: { id, orgId } }) — never findUnique({ where: { id } }). Contact and ProductSupplier mutations verify ownership by joining through the parent entity (supplier → orgId, product → orgId) before any write.
Section 04

Supplier Management Flow

Suppliers are the root entity — Contacts, Documents, and ProductSupplier links all hang off them. Delete is a soft-delete (status → INACTIVE) to preserve audit history. The list endpoint supports multi-filter querying.

Section 05

Document Lifecycle Flow

Documents pass through a defined status workflow. The reviewer identity is always derived from the verified JWT — it cannot be supplied via the request body. File uploads use multer memory storage; filenames are UUID-based (unguessable). File size is capped at 20 MB.

Section 06

Product & Traceability Flow

Products map to multiple suppliers via the ProductSupplier join table with a role. A garment can have a CUT_AND_SEW factory, a FABRIC_SUPPLIER mill, a YARN_SUPPLIER spinner, and a TRIM_SUPPLIER — all linked. SKU is unique per organisation. Delete is soft (DISCONTINUED).

Deletion strategy

EntityDelete typeMechanismReversible?Reason
SupplierSoftstatus → INACTIVEYesPreserve document + contact history

Audit trail principle: All compliance-relevant entities (Documents, Suppliers, Products, DocumentTypes) use soft deletes. Only relationship tables (Contacts, ProductSupplier links) use hard deletes. This ensures that even after a supplier or product is deactivated, its complete document history and approval audit trail remain queryable for regulatory purposes.
